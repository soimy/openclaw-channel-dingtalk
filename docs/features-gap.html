现状（精确版）

解析层已经能处理任意 actionId：src/card-callback-service.ts:76-78 的 extractCardActionId 从 cardPrivateData.actionIds[0] 取出 string，没有任何 whitelist。src/gateway/channel-gateway.ts:347-395 把 callback 路由到三个出口：

关键限制：显式反馈学习只由 AI Card 按钮点击触发 recordExplicitFeedbackLearning（src/gateway/channel-gateway.ts:353）的唯一入口是 AI Card 上 feedback_up / feedback_down 按钮的回调（TOPIC_CARD）。src/feedback-learning-service.ts 内部没有任何 reaction / 入站事件相关代码——它是一条 button-callback 直连 learning store 的闭环。markdown / text 回复路径上没有可承载这些按钮的组件，所以非 Card 回复完全不参与显式反馈学习；隐式学习走 inbound 文本 pattern 匹配（NEGATIVE_SIGNAL_PATTERNS 等）是另一条独立路径。

按钮模板也不只在 src/card/card-template.ts 的字符串里——docs/assets/card-template-v2.json 有动态 ButtonGroup 模板源，配合 cardData 的 variable 渲染。

已有能力

  • 能发 AI Card（src/card-service.ts:845 createAndDeliver）
  • Card 模板里能放按钮（src/card/card-template.ts + docs/assets/card-template-v2.json 动态 ButtonGroup 模板源）
  • Stream 能收到卡片点击回调（TOPIC_CARD /v1.0/card/instances/callback）
  • 能解析任意 actionId（src/card-callback-service.ts:76-78 无 whitelist）
  • 已硬编码处理 3 个 actionId：feedback_up / feedback_down（写 learning service）+ btn_stop（owner-only stop）

真正缺的 6 件事

  1. 通用 action registry：当前没有一张注册表把 actionId → handler 关联起来（"approve 交给谁处理 / retry 交给谁处理 / select_model 交给谁处理"）；当前是代码里硬判断几个 actionId。
  2. 统一 dispatcher：收到 callback 后没有统一分发层。当前路径是"先特殊处理 feedback，再丢给 handleCardAction()，而 handleCardAction() 只认 btn_stop"（src/gateway/channel-gateway.ts:347-395）。
  3. OpenClaw 通用 interactive / approval 接入：其它 channel 通过 plugin 级 approvalCapability 字段（openclaw/src/channels/plugins/types.plugin.ts:86 approvalCapability?: ChannelApprovalCapability，类型在 openclaw/src/channels/plugins/types.adapters.ts:640）把 approval / 按钮 / 选择器接到通用交互运行时；Discord 实例参考 openclaw/extensions/discord/src/channel.ts:354 approvalCapability: getDiscordApprovalCapability() + 工厂在 openclaw/extensions/discord/src/approval-native.ts。src/channel.ts:40的 dingtalkPlugin 没声明这个字段，agent 没法自然发"批准 / 拒绝 / 继续 / 取消"等通用交互。
  4. 通用组件渲染：agent / runtime 如果给一个抽象 interactive payload，钉钉这边不能自动渲染成 AI Card 按钮 / 组件。当前按钮更像是卡片模板里写死的固定能力，不是通用输出能力。
  5. 权限与上下文绑定：通用交互需要知道（a）这个按钮属于哪次 agent run；（b）谁可以点；（c）点的人是不是原发起人；（d）过期了怎么办；（e）重复点击怎么办。btn_stop 有局部的 owner 校验（src/card/card-action-handler.ts:30-41），没有普适化。
  6. 统一交互状态模型：用户点按钮后，通用框架应能更新原卡片状态（"已批准 / 已拒绝 / 已过期 / 已超时"）。当前 feedback 是发确认消息、btn_stop 是停止任务，没有统一的交互状态模型可复用。
一句话总结：钉钉平台支持 card callback，但本仓库还缺"把 callback 变成 OpenClaw 通用交互事件"的那一层。当前是"能收到点按钮"，但还不是"任何 OpenClaw 交互都能在钉钉里自然工作"。

04 · 统一 API 调用调度器（Throttling）

P1 · 健壮性 单接口 40/s/appkey UNCONFIRMED errcode 90018 UNCONFIRMED errcode 88 ≠ 限流（已纠正）
概念分层：throttling vs rate-limit 是两件不同的事
名称谁踩刹车说什么
Rate-limit平台"你一分钟最多调用这么多次，超过我就报错"（HTTP 429 / 含 "rate" 关键字 errcode）
Throttling我们自己"我知道平台有限制，所以排队、降速、合并请求，别一口气全打出去"
本 gap 关注的是 throttling——主动踩刹车。当 rate-limit 已经触发时，throttler 还要负责识别 429 / Retry-After 并做退避。

现状

已有：局部节流（卡片流式场景）
  • cardStreamInterval 默认 1000ms（src/config-schema.ts:139，使用在 src/reply-strategy-card.ts:123）：AI Card 流式更新最小间隔
  • card-draft-controller 自有 throttle，默认 300ms（verbose 模式 50ms，src/card-draft-controller.ts:126）：card draft 提交节流
  • draft-stream-loop 内部 throttleMs（src/draft-stream-loop.ts:26,73,89）：dispatcher 派发节流
这些都是卡片流式场景的局部节流，不是统一的 API 调用调度器。
真正缺的：统一 API 调度器 src/send-service.ts / src/card-service.ts / src/auth.ts / src/inbound-handler.ts（媒体下载）/ src/ack-reaction-service.ts 等所有 OpenAPI 调用都是"即发即走"，没有统一节流层；429 识别与 Retry-After 处理、按 account/endpoint 排队、连续失败短暂熔断、限流指标进入 status/doctor/audit——这些都还没有。
典型受影响场景 机器人同时在 20 个群里回复，或一条 AI Card 正在流式更新——例如：第 1 秒更新卡片 10 次、第 2 秒发送 30 条消息、第 3 秒上传 20 个文件。没有 throttling 时可能：钉钉返回 429 / too many requests → 卡片更新失败 → 消息发送失败 → 失败后继续疯狂重试雪上加霜 → 多账号或多会话一起跑时互相拖垮。

修订后的设计草案（6 类标准实现模式 + 9 项缺口清单）

6 类标准实现模式

  1. 按接口限速：发消息 / 更新卡片 / 上传文件 / 撤回消息 分别控制速率（每个端点一个独立桶）
  2. 按账号限速：每个 DingTalk account 单独一个队列；A 账号被限不影响 B 账号
  3. 合并更新：AI Card 流式更新不应每个 token 都调 API；多个小更新合并成一次（现有 cardStreamInterval / card-draft-controller / draft-stream-loop 三处局部 throttle 已是这一类的雏形，但只覆盖 card 场景）
  4. 遇到 429 后退避：平台说"太快了"就等几秒再试；若响应 header 有 Retry-After，优先按它等，不是固定 5s
  5. 短暂熔断：连续失败时暂停某类 API 一段时间，防止一直打失败请求；连续成功后逐步恢复
  6. 日志 / 指标：记录"哪个接口被限流、排队多少、延迟多少、丢了多少"，进入 status snapshot（#07）、doctor 报告（#02）、audit log（#02 throttle.suspended / throttle.recovered）

  • ⑨ 限流状态进入 status snapshot / doctor 报告 / audit log

限流识别（不依赖具体单一 errcode）

  • HTTP 429 → 限流，按 Retry-After 等待（无则保守 5s）
  • errcode 含 "rate" / "limit" / "frequency" 关键字 → 限流

06 · 持久化 Dedup（跨重启 / 长 TTL / 多实例）

P2 · 健壮性 事件订阅不 ACK 会重投 DOCUMENTED（protocol 原文） 机器人 robot fire-forgot DOCUMENTED（protocol 原文） SDK 通用 60s 重试注释 CONFIRMED（client.d.ts:97）
什么是 dedup？为什么要做？ 插件可能在两种场景下重复看到同一条入站：(a) 钉钉把事件订阅推给插件 → 插件处理慢了 / ACK 异常 / 连接断了 → 钉钉按 protocol "不回复 ACK 触发服务端超时重新推送"语义重推同一事件；(b) 多实例 / 重启窗口里同一消息被多个进程或新旧进程看到。如果插件不去重，机器人可能重复处理。所以插件要记住"robotKey + msgId / eventType + eventId 这条已经处理过"，下次同样的来就跳过。

现状

已有：内存 60 秒 dedup（src/dedup.ts:1-67） 
const processedMessages = new Map<string, number>();
const MESSAGE_DEDUP_TTL = 60000;
const MESSAGE_DEDUP_MAX_SIZE = 1000;
第 1 行 DEDUP_NAMESPACE_POLICY = "memory-only" 显式声明策略。能挡住同进程 60 秒内的重复推送。
4 种当前挡不住的场景
  1. 进程重启：记录全没了；重启前后窗口内的重推会被当成新消息
  2. 60 秒外的延迟重推：平台延迟超过 TTL 后再推同一条，会被再次处理
  3. 多实例 / 多 worker 部署：A 实例处理过，B 实例不知道（HA / 蓝绿 / 灰度都会触发）
  4. 跨进程的灰度升级：旧 process 没 ack、新 process 收同事件

平台能力评估

DOCUMENTED 事件订阅：不 ACK 会重投，需要 dedup 钉钉 Stream 协议对事件订阅 ACK 语义的原文：
如果不回复 ACK，会触发服务端的超时重新推送，以及客户端对相同事件的重复处理。
这是事件订阅必须 dedup 的官方依据——服务端会重推，客户端必须自身识别重复事件。
DOCUMENTED 机器人消息：当前是 fire-forgot，不因超时重推 同一 Stream 协议对 robot callback 的原文备注：
备注：机器人消息当前是 fire-forgot 模式，不会因为网络超时而重复推送。
以及对 ACK 用途的分层说明：
ACK 主要用途…2. "事件订阅"之外的推送，ACK 仅用于问题诊断和分析。
所以"事件订阅"与"事件订阅之外（robot callback / card callback）"的可靠性语义是官方文档明确区分的，不是社区猜测。
CONFIRMED SDK 通用 60s 重试注释（与 protocol 分层语义并行存在） 本地 dingtalk-stream@2.1.4 的 node_modules/dingtalk-stream/dist/client.d.ts:97-108 对 socketCallBackResponse 的 JSDoc：
stream 模式下，服务端推送消息到 client 后，会监听 client 响应，如果消息长时间未响应会在一定时间内 (60s) 重试推消息，可以通过此方法返回消息响应，避免多次接收服务端消息。
这是 SDK 通用层的描述（**不区分** robot / event / card）。SDK 提供了统一的 ACK/response 通道，但官方 protocol 对 event subscription 与 robot callback 的可靠性语义做了区分——event 是真的会重推，robot 当前不重推。两个证据并不矛盾，是不同层次。
对标实证 openclaw 内对标修正：Feishu 用 createPersistentDedupe（openclaw/extensions/feishu/src/dedup.ts:27-30），磁盘 JSON 落盘——这是真正的持久化 dedup。Discord 用 createClaimableDedupe（openclaw/extensions/discord/src/monitor/inbound-dedupe.ts:1,9），不是纯内存。Telegram 的入站 update dedup 是 createDedupeCache 内存 5 分钟（openclaw/extensions/telegram/src/bot-updates.ts:71）；之前误引的 message-cache.ts 是消息上下文持久化，不是 update dedup。Feishu / Discord 的实现可作 #06 / #11 落地参考。
持久化 dedup 的 standalone 价值 持久化 dedup 应作为独立可立项的能力，而不条件性绑定 #11，理由：
  • 事件订阅按 protocol 原文必定会重投（"不回复 ACK，会触发服务端的超时重新推送"）—— 这是 dedup 的第一硬需求
  • "机器人 robot callback 当前 fire-forgot 不重推"也是 protocol 原文，但仅指网络超时层——多实例 / HA / 灰度部署、客户端进程重启窗口、平台未来策略变更，都不在这条 fire-forgot 承诺的覆盖面内
  • 多实例 / HA / 灰度部署场景下，持久化 dedup 是必需的（与 robot ack 是否重推无关，是分布式一致性问题）
  • 事件订阅类（recall / read）确实会重投，且 #11 已确认要做，那一层也需要 dedup（共用同一抽象即可）
核心受益面是事件订阅 dedup（DOCUMENTED 重投）+ 多实例部署 + 重启窗口三个场景。

持久化 dedup 长什么样

把"已处理消息"的记录写到文件 / 数据库 / 共享存储里。最小化结构：

{
  "dingtalk:robotA:msg123": {
    "processedAt": 1710000000000,
    "expiresAt": 1710086400000
  }
}

带来的能力：

  • 重启后仍能去重
  • 平台延迟重试也能挡住（超过当前 60s TTL）
  • 多实例可以共享状态
  • 可以设置更长 TTL（例如 24 小时）

权衡 / 约束（不是越长越好）

  • 有 TTL：长 TTL 会让存储膨胀；推荐 24h 起步，可配置
  • 定期清理：到期记录自动 GC，避免无界增长
  • key 稳定：必须用平台保证唯一且稳定的字段（如 robotCode + msgId），不要混入时间戳 / 内容 hash
  • 不保存敏感正文：只保存消息 ID / 状态 / 时间戳，不存内容
  • 失败处理：处理失败时不能误标"已完成"——只有处理成功才写 dedup；失败的允许重试
  • 跨实例存储：单实例可用文件 ndjson；多实例需要共享存储（filesystem 挂载 / Redis / 等），按部署形态选

设计草案

  • 存储格式：storePath 下 dedup/{accountId}-messages.ndjson（机器人消息）+ dedup/{accountId}-events.ndjson（事件订阅，#11 同步落地）；append-only ndjson，启动时全量加载到内存 Map
  • 层级：内存 Map（hot path）→ ndjson 落盘（cold / restart recovery）；写操作 fire-and-forget，不阻塞主线程
  • TTL：默认 24h，可在配置里调；启动时跳过已过期条目
  • 清理：定时 GC（默认 1h 一次）+ 启动时压缩 ndjson
  • key 选取：消息用 {robotCode}:{msgId}；事件订阅用 {eventType}:{eventId}
  • 写入时机：仅在处理成功后写；失败的不写（允许下次重试）
  • 多实例：v1 单实例（文件 ndjson）；v2 多实例（按部署形态加入 shared store，例如挂载同一卷或接入 Redis）

08 · 启动 Stagger（工程防风暴，不是平台硬要求）

P3 · 按需开（默认关） "钉钉网关连接频率限制" WRONG（已纠正） 现有 reconnect jitter 已实现 CONFIRMED
什么是启动 stagger？ 多个账号 / 连接启动时不要同时冲出去，而是错开几秒一个个启动。例子：10 个钉钉账号一起启动时—— 
// 无 stagger（同时冲）
10:00:00  acc1  连 Stream
10:00:00  acc2  连 Stream
10:00:00  acc3  连 Stream
...

// 启用 stagger（错开 5s）
10:00:00  acc1  启动
10:00:05  acc2  启动
10:00:10  acc3  启动
...
同时启动容易瞬间打爆 5 类资源：① 获取 access token；② 获取 Stream ticket；③ 建 WebSocket；④ 注册 callback；⑤ 各种初始化探活。
启动 stagger 主要防什么
  • 服务重启时多个账号一起抢 API
  • 多实例同时部署时形成启动风暴
  • 钉钉 Stream ticket 这类短时凭证被并发消耗（ticket 90s 单次有效）
  • 同一个 clientId 被重复启动（这是官方硬约束的违反风险）
  • 瞬时 429 / 连接失败

现状

已有：reconnect jitter（单账号场景） src/connection-manager.ts:243-259 的 calculateNextDelay 已实现指数退避 + jitter（断线重连场景），但没有覆盖多账号启动场景。
真正缺的：多账号启动错峰 钉钉插件当前没有多账号启动错峰机制。单账号问题不大；多账号 / 多实例部署时可能出现启动风暴或重复连接风险（5 类资源争抢见上方）。

平台能力评估

WRONG "钉钉网关有连接频率限制" / "单 IP 30s 封禁" —— 找不到任何官方依据 官方 Stream 协议文档没有提"网关连接频率限制"。之前写"多 Stream client 可以做负载均衡"也不准确——card-examples README 实际给的是"线上线下可用不同 client-id 隔离"（隔离/调试用途，非负载均衡）。准确表述应为"官方允许跨 client-id 多连接（用于隔离/调试），但每个 client-id 仍是单连接约束"。"启动 stagger 是平台硬要求"这个定位仍然是错的。
CONFIRMED 钉钉官方唯一真实的 stream 约束 dingtalk-card-examples README:
请务必保证一个 client-id 同一时间只启动一个 Stream 服务。
官方 Stream 协议:
ticket 的有效期为 90 秒，且只能用来建立一条连接。
这两条都不是"频率限制"，是"同 client-id 唯一性约束"。

修订后的定位：工程防风暴措施

stagger 应被定位为工程预防措施 / 稳定性工程，不是平台硬要求也不是用户功能：

  • ① 防 token 接口共享配额：同 process 内多 account 同时拉 token 可能挤兑 OpenAPI 配额（与 #04 throttler 互补）
  • ② 防 Stream ticket 并发消耗：ticket 有效期 90s 且只能用一次（官方明示），并发启动可能让某些 account 拿到过期 ticket
  • ③ 防进程内瞬时资源占用：多 account 同时建 WS handshake 占瞬时 CPU / network
  • ④ 防 callback 注册风暴：同时注册多个 TOPIC_ROBOT + TOPIC_CARD 监听
  • ⑤ 多实例部署时降低同时启动概率：部署形态外的并发由 stagger 内部错峰缓冲
  • 不是必需：如果只有 1-3 个账号，stagger 收益微小，可以不做（默认关闭）

对标实证：openclaw Discord 用 DISCORD_ACCOUNT_STARTUP_STAGGER_MS = 10_000（openclaw/extensions/discord/src/channel.ts:85）做账号启动错峰；Telegram / Feishu / Slack 未实现 stagger。

设计草案（保留但降级）

  1. 账号级 stagger：账号之间 500ms 错峰 + jitter ±200ms，默认关闭，配置开启
  2. Reconnect jitter：单账号已有；继续保留
  3. 同 client-id 检测：这是平台硬约束的真正应对——检测同 client-id 在同 process 启动多次，拒绝
